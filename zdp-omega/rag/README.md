# ZDP-╬® RAG ÔÇö making the protocol runnable by a small model

This turns the 34-file ZDP-╬® protocol into something a weak or small model
(the target is ~4B parameters, but this works for any size) can execute
reliably, one atomic step at a time, without ever holding the whole protocol
in its context window ÔÇö and without ever being able to silently skip part of
it.

The knowledge base indexed here is **the protocol itself** (the 26 phase
files + appendices + master protocol + boot file). It is built **once**. The
resulting tool is then pointed at **any codebase you want audited** ÔÇö the
model uses its own file-reading tools to inspect that codebase, and calls
this RAG tool only to find out what the protocol requires at each step.

```
rag/
Ôö£ÔöÇÔöÇ README.md                     (this file)
Ôö£ÔöÇÔöÇ build_index.py                the indexer ÔÇö run once, or after editing the .md source files
Ôö£ÔöÇÔöÇ zdp_rag.py                    the runtime tool the model actually calls
Ôö£ÔöÇÔöÇ SMALL_MODEL_SYSTEM_PROMPT.md  paste this as the system prompt for the model driving the audit
Ôö£ÔöÇÔöÇ index/                        generated by build_index.py ÔÇö the shared, read-only knowledge base
Ôöé   Ôö£ÔöÇÔöÇ chunks.jsonl              every retrievable unit, portable JSON Lines
Ôöé   Ôö£ÔöÇÔöÇ flight_plan.json          the entire audit flattened into one strict ordered list of steps
Ôöé   ÔööÔöÇÔöÇ zdp_index.sqlite          the same data + FTS5 full-text search
ÔööÔöÇÔöÇ state/                        generated per audit run ÔÇö one small JSON file per codebase you audit
    ÔööÔöÇÔöÇ <run-name>.json
```

## Why this design (read this before assuming it's under-built)

**No embeddings, no model download, no pip install.** Everything here is
Python 3.9+ stdlib: `re`, `json`, `sqlite3`, `argparse`. That's a deliberate
choice, not a shortcut:

- This is meant to sit next to a small local model with zero setup friction,
  possibly offline, possibly on modest hardware. Requiring a sentence-transformer
  download defeats that purpose.
- The corpus is dense with exact technical vocabulary and IDs (`R10.1`,
  `CWE-89`, `WCAG 2.2 SC 2.5.8`, `ASVS v5.0.0-1.2.2`) ÔÇö the kind of retrieval
  where **exact and keyword-ranked search (SQLite FTS5, BM25-style ranking)
  usually beats semantic embeddings**, not the other way around. Embeddings
  shine on fuzzy natural-language similarity; this corpus rewards precision.
- If you later want semantic/fuzzy search on top of this (e.g. "find
  something like a race condition but for caches"), `chunks.jsonl` is a
  plain, portable list of `{chunk_id, text, ...}` records ÔÇö bolt any
  embedding pipeline onto it without touching the rest of the system.

**The flight plan, not the chat, is what prevents missing something.** The
model is never trusted to remember what it already checked or to decide what
order to work in. `build_index.py` parses every single rule (`- **R10.1**
...`), every falsification-procedure bullet, every exit-gate checklist item,
and every numbered setup step (`### Step 1 ...`) out of the source documents
mechanically, in document order, phase 00 ÔåÆ 25, and writes them to
`flight_plan.json` as one flat, ordered list. `zdp_rag.py next` always
returns the first one that has no recorded verdict yet. There is no path for
a step to be silently skipped ÔÇö it either gets a verdict or it stays first
in line forever.

This was validated, not assumed: after building the index, every phase's
extracted rule IDs and gate-checklist counts were cross-checked against an
independent regex sweep of the raw files. All 26 phases matched exactly
(current build: **826 flight-plan steps** ÔÇö 509 rules, 194 exit-gate items,
109 falsification procedures, 14 setup steps ÔÇö across 867 total indexed
chunks, including reference material from the appendices).

## Build the index

```bash
cd rag
python3 build_index.py
```

Re-run this any time you edit the `.md` source files (new rules, edited
appendices, a 27th phase, whatever). It fully rebuilds `index/`; it does not
touch anything in `state/`, so in-progress audit runs are unaffected unless
you actually renumber or delete rules they already reference.

## Run an audit

```bash
cd rag
python3 zdp_rag.py --run my-audit init --codebase "~/projects/my-app"

# the loop ÔÇö repeat until "No unresolved steps remain"
python3 zdp_rag.py --run my-audit next
# ... go look at the actual codebase ...
python3 zdp_rag.py --run my-audit complete <step_id> --verdict PASS --evidence "..."

python3 zdp_rag.py --run my-audit status
```

`--run <name>` lets you keep several audits (different codebases, or repeat
audits of the same one over time) alive against one shared protocol index ÔÇö
just use a different run name per audit.

## The five commands

| Command | Purpose |
|---|---|
| `init --codebase "..."` | label a run |
| `next [--phase N]` | the next unresolved step, in full, mind-tagged, ready to act on |
| `complete STEP_ID --verdict V --evidence "..."` | record a verdict; `FAIL` requires `--severity`; every verdict requires `--evidence` |
| `search "query" [--kind K] [--phase N]` | keyword search across the whole corpus ÔÇö does not advance anything |
| `get ID` | exact fetch by `step_id`, `chunk_id`, or `rule_id` |
| `status` | progress by phase, open findings by severity, a loud banner for any open P0 |
| `findings [--severity P0]` | dump recorded findings as JSON (maps directly onto the `FND-*` schema in `APPENDIX_TEMPLATES.md` ┬ğC.3) |

`complete` enforces two guardrails mechanically, taken straight from
`AGENT_BOOT.md`: a `FAIL` verdict is rejected without `--severity` (Prime
Directive #8, "severity is law"), and every verdict is rejected without
`--evidence` (Prime Directive #2, "evidence or it didn't happen"). A small
model cannot accidentally produce an unscored or unsupported finding ÔÇö the
tool errors out and tells it exactly what's missing.

## Wiring it to a model

This is intentionally a plain CLI, not a bespoke framework, so it drops into
whatever you're already using:

- **Local model via function-calling / tool-use** (Ollama, llama.cpp
  server, etc.): expose `next`, `complete`, `search`, `get`, `status` as
  four or five tool functions that shell out to `zdp_rag.py`. Give the model
  `SMALL_MODEL_SYSTEM_PROMPT.md` as its system prompt.
- **Agentic coding tool with shell access** (Claude Code and similar): the
  model can just run the commands directly in a terminal. Same system prompt
  applies.
- **No tool-use at all, pure text loop**: a driver script calls `next`,
  pastes the output plus "look at the codebase and answer with a verdict" to
  the model, parses the model's answer, calls `complete`, repeats. The
  protocol enforcement lives entirely in the tool either way.

## What this does *not* do

- It does not read the codebase being audited for you. That's the model's
  own file/shell tools, per step, guided by what `next` tells it to look for.
- It does not decide which of the 26 phases apply to a given codebase (e.g.
  Phase 21 AI/ML on a codebase with no models). Every step still gets a
  verdict ÔÇö the correct one for "doesn't apply" is `N/A`, which is a
  first-class, fast-to-give verdict, not a skip.
- It is not a semantic/fuzzy search engine (see above) ÔÇö if you need that,
  extend `chunks.jsonl` with your own embedding index; the data's already
  portable for it.
- It does not replace the human checkpoints `00_MASTER_PROTOCOL.md` ┬ğ7
  requires (TIMELINE conflicts, SIL-4 contract changes, requirement-altering
  P0 fixes). `BLOCKED` is exactly the verdict for "this needs a human," and
  `status` surfaces every BLOCKED step so nothing gets stuck silently.
